# Cal.com Clone

A full-stack, responsive scheduling and booking web application that mimics the design and user experience of Cal.com. This project allows users to set up availability, create dynamic event types, and provides a sleek public interface for clients to book meetings securely without double-booking.

## 🚀 Tech Stack

**Frontend:**
- **React.js (Vite)**: Fast, modern SPA framework.
- **Tailwind CSS v4**: Utility-first CSS framework for replicating Cal.com's beautiful aesthetics.
- **React Router DOM**: Client-side routing with protected dashboard routes.
- **Lucide React**: Clean, consistent SVG icons.
- **Axios**: HTTP client equipped with interceptors for JWT injection.
- **Date-fns**: Robust date manipulation for the calendar logic.

**Backend:**
- **Node.js & Express.js**: Lightweight and fast REST API architecture.
- **MySQL (mysql2/promise)**: Relational database to ensure transactional safety (preventing double bookings) and complex join queries.
- **JWT (JSON Web Tokens)**: Secure stateless session management for admin dashboard access.
- **Bcrypt.js**: Cryptographic password hashing to keep user credentials secure.

## ✨ Core Features Implemented

1. **Event Types Management**: 
   - Admin can Create, Edit, Delete, and List event types (title, description, duration, and slugs).
   - Each event automatically generates a unique public booking endpoint `/:username/:slug`.
2. **Availability Settings**:
   - Toggle specific days of the week on/off.
   - Set custom start times and end times per active day.
   - Dynamic Custom Timezone Selector.
3. **Public Booking Page**:
   - Polished calendar UI allowing guests to select dates.
   - Dynamic available time slots are generated by querying the backend to subtract already booked meetings from the host's specified standard working hours.
   - Clean, frictionless booking confirmation payload.
4. **Bookings Dashboard**:
   - List view for the host to monitor upcoming and past meetings.
   - Cancellation capability with instant visual feedback.
5. **Secure Authentication**:
   - JWT integration locking the dashboard behind an authenticated `admin@calclone.com` gate.
   - Auto-verifying Router guards.
6. **Responsive Design**:
   - Complete mobile-first layouts with collapsible sliding sidebars matching Cal.com’s exact structural patterns.

## 🛠️ Setup Instructions

### Prerequisites
- Node.js (v18+)
- MySQL Server (running locally)

### 1. Database Setup
1. Create a MySQL database named `cal_clone`.
   ```sql
   CREATE DATABASE cal_clone;
   ```
2. Navigate into the `server` directory and configure your `.env` file:
   ```env
   PORT=5000
   DB_HOST=localhost
   DB_USER=root
   DB_PASSWORD=your_mysql_password
   DB_NAME=cal_clone
   JWT_SECRET=supersecretkey123
   ```
3. Initialize the database schema and seed the default user by running the built-in script:
   ```bash
   cd server
   npm install
   node scripts/initDB.js
   node scripts/migrateAuth.js
   ```

### 2. Running the Backend
```bash
cd server
npm start
```
*The server will start on port 5000.*

### 3. Running the Frontend
```bash
cd client
npm install
npm run dev
```
*Vite will start the client locally. The frontend is securely proxying `/api` requests to the Backend via Vite's proxy config.*

## 🔐 Default Admin Credentials

- **Email/Username**: `admin@calclone.com`
- **Password**: `password`

## 🧩 Assumptions & Design Decisions
- **Single-User Scope**: While the database supports a `users` table, the frontend currently centers around the logged-in user dashboard context. Public pages simulate the route `/:username/:slug` but default to tracking the seeded admin's configurations.
- **Timezone Normalization**: Time calculations for bookings are treated contextually, allowing the host to select what timezone they are operating within for clear calendar rendering.
- **No Extra Dependencies**: Hand-built calendar structures using fundamental `date-fns` utility blocks over bloated 3rd-party calendar libraries to maximize CSS parity with Cal.com.
- **Deletion Rules**: Canceling a meeting invokes a soft 'status update' (`status = "cancelled"`) rather than a hard DB delete, allowing for detailed historical audit logs for the host in the future.
